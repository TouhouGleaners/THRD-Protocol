# 权限矩阵与资源分配 (Access Control)

在 **THRD | 拾遗梦茶馆**，权限分配遵循 **“能力对等”** 与 **“职责清晰”** 的原则。权限不是身份的象征，而是维护能力的背书。

---

## 1. 维护者权限等级

我们根据维护者的 [技术准入标准](../skills/tech-stack.md)，将内部协作权限划分为三个等级：

### 等级 L1：基础维护者 (Basic Maintainer)
- **准入要求**：掌握 Git/Markdown 基础协作流程，完成首个准入 PR。
- **核心权限**：
    - `THRD-Protocol` 仓库的 Issue 管理与提案参与权。
    - 组织内部文档的日常维护与修订权。
- **职责**：确保组织规章与 SOP 的准确性。

### 等级 L2：业务维护者 (Specialized Maintainer)
- **准入要求**：在满足 L1 基础上，通过 [Python 或 Vue 3 的选修路径](../skills/tech-stack.md) 考核。
- **核心权限**：
    - **Python 方向**：TMA 仓库、Danmaku Sender 仓库的维护与非核心代码合并权。
    - **Vue 3 方向**：官网前端组件库、TMA 监控看板代码仓库的维护权。
- **职责**：负责具体业务模块的迭代与质量把控。

### 等级 L3：核心维护者 (Core Maintainer)
- **准入要求**：具备全栈审计能力，对项目安全性与整体架构负责。
- **核心权限**：
    - GitHub 组织（Organization）管理员权限。
    - 核心业务逻辑的最终合并（Merger）与发布裁决权。
- **职责**：负责全局风险控制与技术选型。

## 2. 状态流转：休眠与复苏

我们理解每位维护者在现实生活中都有学业、工作等高峰期，因此 THRD 实行灵活的**休眠保护机制**，旨在平衡“资产安全”与“成员归属感”。

### 状态定义
- **活跃 (Active)**：持续参与仓库维护、评审或社区讨论。
- **休眠 (Dormant)**：连续超过 180 天未在 GitHub 组织内留下任何协作轨迹（Commit/Review/Issue/Comment）。

### 机制说明
1.  **休眠保护**：若维护者进入休眠状态且未提前报备，其 L2 及以上等级的写权限将暂时调回至 L1 级别。
    - *初衷*：这仅仅是为了在成员“不看家”时，降低账号被盗导致的组织资产风险，并非对成员贡献的否定。
2.  **一键复苏**：休眠成员回归后，**无需重新考核**。只需在 Issue 中打个招呼或提交一个简单的文档 PR，即可由核心维护者恢复原有权限等级。
3.  **请假免停**：若因现实生活繁忙需暂时离岗，只需提前告知，权限将一直保留。
