# 权限矩阵与资源分配 (Access Control)

在 **THRD | 拾遗梦茶馆**，权限分配遵循 **“能力对等”** 与 **“职责清晰”** 的原则。权限不是地位的象征，而是维护职能的背书。我们通过 GitHub 的分支保护机制，确保所有变更均经过评审与审计。

---

## 1. 核心红线
!!! danger "严禁直接提交"
    除了紧急状态下的核心维护者外，**严禁任何人直接向仓库主分支 (main) 提交代码或文档**。所有的修改动作必须通过 **Pull Request (PR)** 进行，并在通过评审后由核心维护者合并。

---

## 2. 职能路径与授权标准

我们根据维护者的 [选修路径](../skills/tech-stack.md) 分配相应的协作权限：

### **运营维护权限 (Operational Access)**
*针对侧重内容审核、社群治理与宣发的维护者。*

- **GitHub 资产**：对 `THRD-Protocol` 及各业务仓库拥有 **Issue 管理权** 与 **PR 提交（提议）权**。
- **发布资产**：官网 (WordPress) 后台编辑权限、官号管理协作权。
- **准入红线**：必须通过 [入馆 PR 挑战](../action-guide/onboarding.md)。

### **技术维护权限 (Technical Access)**
*针对侧重工具开发、数据抓取与环境维护的维护者。*

- **GitHub 资产**：在“运营权限”基础上，额外获得对应业务仓库（如 TMA, Danmaku Sender）的写权限与非核心代码合并权。
- **技术资源**：业务相关的 API Tokens 访问权、部署逻辑的审核权。
- **准入红线**：需达到 Python 或 Vue 3 的 [最小准入标准](../skills/tech-stack.md)。

### **核心维护权限 (Core Access)**
*针对具备全栈审计能力、负责组织架构安全的维护者。*

- **组织管理**：GitHub 组织 (Organization) 管理员权限、所有核心仓库的最终合并确认权。
- **外部对接**：负责与外部 Host（部署方）进行最终的技术发布对接。
- **准入红线**：全栈维护能力与长期稳定的协作信誉。

---

## 3. 状态流转：休眠保护机制

为了平衡“资产安全”与“成员归属感”，THRD 实行灵活的休眠保护机制。

- **休眠定义**：连续超过 **180 天** 未在 GitHub 组织内留下任何协作轨迹（Commit/Issue/Comment）。
- **休眠处理**：相关账号的 **业务代码仓库（如 TMA, Danmaku Sender）写权限** 将暂时关闭，仅保留 **基础文档（Protocol）的维护权限**。
    - *目的*：这仅仅是为了在成员长期不在岗时，降低账号被盗导致的生产环境风险。
- **一键复苏**：休眠成员回归后，**无需重新考核**。只需提交一个 PR 或在 Issue 留言说明，即可由核心维护者手动恢复原有权限。
