# 决策流与提案管理 (Decision Making)

在 **THRD | 拾遗梦茶馆**，我们通过“异步协作”与“文档对齐”来确保决策的透明与可追溯。我们拒绝碎片化的即时通讯决策，所有的正式决定必须经历 **Issue 讨论** 到 **PR 定稿** 的转化。无论你是运营型维护者还是技术型维护者，都通过同一套流程参与组织决策。

!!! note "适用范围说明"
    - **本文档适用于**：组织级制度变更、重大项目立项、跨职能协作方案、或独立活动提案。
    - **日常任务提醒**：对于具体的代码迭代、已知 Bug 修复或纯文字排版优化，请遵循 [开发与维护工作流](../action-guide/dev-workflow.md) 直接进入任务执行。

---

## 1. 决策工具的分工

为了兼顾讨论的自由度与结果的严肃性，我们对工具进行明确分工：

| 工具 | 对应阶段 | 作用 |
| :--- | :--- | :--- |
| **GitHub Issue** | **初步想法 / 讨论** | 用于非结构化的讨论。例如：“我们要不要办周年庆？”“某个脚本是不是该重构？” |
| **GitHub PR** | **正式方案 / 评审** | 用于结构化的定稿。针对具体的文档修改或提案进行逐行评审与修正。 |
| **Git Merge** | **决策生效 / 归档** | 核心维护者合并 PR。合并的那一刻，代表决策正式生效，并成为组织历史档案。 |

---

## 2. 提案分类与路径

### A. 协议修正 (Protocol Modification)
- **对象**：修改现有的管理制度、技术标准、权限矩阵等。
- **操作**：直接对 `docs/` 下的已有文件发起 PR。
- **意义**：这是在更新茶馆的“宪法”。

### B. 独立提案 (Proposals)
- **对象**：临时活动（周年庆）、新项目立项、外部合作。
- **操作**：在 `docs/proposals/` 目录下新建一个 Markdown 文件。
- **意义**：这是在茶馆的“记事本”上写下新的执行计划。

## 3. 标准提案流程 (RFC 流程)

无论是修改现有制度，还是发起像“周年庆”这样的临时活动，统一遵循以下流程：

### 第一步：发起 Issue（吹风阶段）
任何维护者都可以发起 Issue。

- **目的**：试探大家的意向，收集零散的建议。
- **操作**：使用 `[讨论]` 或 `[意向]` 标签。

### 第二步：提交 PR（起草阶段）
当想法成熟，发起人需提交一个 Pull Request：

- **运营维护者**：可直接使用 GitHub 网页端的“铅笔”图标进行起草。
- **技术维护者**：建议在本地修改后通过命令行提交。

### 第三步：评审与修订（拉锯阶段）  
这是最关键的一步。其他维护者在 PR 的具体行下发表意见。

- **精准对线**：不要说“我觉得方案不行”，而要针对具体条款说“这个方案的时间可能不够”。
- **持续更新**：发起人根据评审意见，直接修改该 PR 中的文件内容，直至达成共识。

在此期间，发起人根据建议不断完善方案，运营人员可重点关注业务逻辑，技术人员重点关注实现成本。

### 第四步：裁决（生效阶段）
由核心维护者对 PR 进行**最终裁决**：

- **批准**：点击 Squash and Merge。方案正式转化为茶馆的执行指令或历史档案。
- **否决**：核心维护者有权拒绝合并不符合标准的提案，但必须给出基于本 Protocol 的书面解释。当核心维护者人数超过 3 人时，重大否决需经核心维护者内部多数通过。

!!! info "关于审计权"
    核心维护者的合并权本质上是质量与安全审计。核心维护者应尽量尊重集体共识，仅在提案严重偏离技术标准、法律底线或组织精神时行使否决权，并需在 PR 评论中详细说明理由。

## 4. 为什么必须走 PR 流程？

!!! tip "管理即代码的优势"  
    1.  **所见即所得**：无论 Issue 里聊了多少页，文件里永远只展示**最新的、最终的**执行方案。
    2.  **版本可回溯**：通过 Git History，我们可以清晰看到方案在讨论过程中每一个版本的演变（谁在哪天改了哪一个字）。
    3.  **强制理性**：撰写 Markdown 的过程会迫使发起人理顺逻辑，过滤掉情绪化和不切实际的冲动。

---

## 5. 紧急例外情况

!!! danger "紧急避险"
    对于服务器故障、数据泄露等需要即时响应的事务，核心维护者有权先在群内决策并执行。  
    **强制要求**：在事故处理完毕后，必须在 `docs/action-guide/incident-report.md` 中补齐复盘记录并归档，否则该决策不具备长期效力。
