# 开发与维护工作流 (Development Workflow)

在 THRD，我们像维护代码一样管理组织。为确保所有改动可被审计、可被回溯，组织内的一切变更必须通过 **Pull Request (PR)** 流程。

!!! note "适用范围说明"
    - **本文档适用于**：具体的开发与维护任务（代码修改、Bug 修复、文档微调、功能实现）。
    - **重大事项提醒**：若涉及组织级制度变更、新项目立项或重大活动决策（如修改本 Protocol、周年庆筹备），请参阅 [决策流与提案管理](../governance/decision-making.md)。

---

## 1. 任务起源：Issue 驱动

除 **`Quick Fix`** 范畴外的所有任务，必须先建立 Issue 作为讨论凭据。

- **标准任务**：提交 Issue -> 标注 `Status: Triage` -> 讨论通过后打上 `Status: Ready for Developer` -> 开始执行。
- **快速修复 (Quick Fix)**：针对错别字、错误的数学计算或极微小的排版修正，可**跳过 Issue 阶段**，直接发起带有 `Quick Fix` 标签的 PR。
- **调研任务**：对于不确定的方案，打上 **`Status: Needs Investigation`**，并在 Issue 中沉淀调研结果。

---

## 2. 环境协作：分支与提交

严禁任何成员（包括核心维护者）向 `main` 分支进行直接推送。

1.  **拉取分支**：基于最新 `main` 创建本地分支。
    - 命名建议：`类型/描述`（如 `feat/labels-docs`）。
2.  **Conventional Commits**：提交信息必须清晰描述意图。
    - 格式：`类型: 简短描述`。
    - 示例：`fix: 修正技术大纲中的路径描述`。

---

## 3. 核心环节：Pull Request (PR)

PR 是决策生效、代码入库的唯一合法路径。

- **自动化联动**：在 PR 描述中包含 `Closes #Issue编号`，以便合并后自动关闭对应任务。
- **本地自检**：文档类修改发起前，必须执行 `mkdocs serve` 确认渲染效果。
- **评审机制**：
    - 所有 PR 必须指派相关维护者进行 Review。
    - **紧急任务 (Bug: Critical)**：若涉及核心服务崩溃或严重数据风险，打上 `Bug: Critical` 标签。此类 PR 具备最高优先级，核心维护者将进行即时审计并合并。

---

## 4. 完结归档：Squash and Merge

1.  **策略**：THRD 统一采用 **Squash and Merge**。分支上所有的琐碎提交将被压缩为一个干净的记录合并入主分支。
2.  **裁决**：合并权仅限核心维护者。合并动作即代表该决策正式转化为组织的执行指令或历史档案。
3.  **清理**：合并完成后，必须删除 GitHub 上的开发分支以保持仓库整洁。
