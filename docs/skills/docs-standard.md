# 文档标准与协作规范 (Documentation Standards)

在 THRD，**文档即规矩**。为了确保组织信息的高效流动与长期存档，所有维护者必须遵循本套文档标准。

---

## 1. 核心审美：工业级极简主义

我们追求结构清晰、逻辑闭环的表达：

- **无压力阅读**：大量使用标题、列表和提示框，拒绝出现“文字墙”。
- **去富文本化**：统一使用标准 Markdown，严禁 Word 式的冗余排版。
- **客观克制**：语气中立，减少模糊词汇（如“可能”、“大概”），确保信息可执行。

---

## 2. 文件管理规范

- **Kebab-case 命名**：文件名统一使用全小写，单词间用连字符 `-` 连接（如 `dev-workflow.md`）。
    - *原因*：URL 友好、跨平台兼容、无需 Shift 键输入。
- **路径对齐**：文件必须存放在对应的逻辑目录下。

---

## 3. 核心语法速查

### 3.1 常用子集
- **标题**：使用 `#` 逐级递减。全篇仅一个一级标题。
- **加粗**：用 `**粗体**` 强调核心结论。
- **链接/图片**：必须使用**相对路径**。例如 `[权限](../governance/access-control.md)`。
- **提示框 (Admonitions)**：使用 `!!! tip/info/danger` 语法突出重点。

### 3.2 关于任务列表 (Checklists) 的禁令
!!! danger "渲染禁令"
    由于 MkDocs 静态渲染对 `- [ ]` 语法的支持不稳定，**严禁**在 `docs/` 目录下的正式文档中使用任务列表。

    - **Issue/PR 讨论**：可以使用，方便进度追踪。
    - **正式文档**：一律改用标准的无序列表（`-`）。

## 5. 排版礼仪 (Typography)

为了保证文档的工业级质感，请遵守以下细节：

1.  **盘古之白**：在中文与英文、数字之间，请手动加一个**半角空格**。
    - *正确*：使用 Python 3.14 进行补档。
2.  **空行原则**：每一个标题、列表、提示框的前后，**必须留一行空格**。
3.  **代码块标注**：所有代码块必须注明语言（python, yaml, bash 等）。

---

## 6. 发起 PR 前的自检事项

在发起 PR 之前，请手动确认以下事项：

- 是否存在大段未拆分的文字？
- 文件名是否为全小写加连字符？
- 提示框前后是否留有空行？
- 图片和链接是否使用相对路径？
- **本地预览 (mkdocs serve) 是否完全正常？**
