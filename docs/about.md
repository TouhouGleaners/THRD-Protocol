# 关于 THRD Protocol

**THRD Protocol**（拾遗梦茶馆管理协议）是“拾遗梦茶馆”实现工业化管理转型的核心基石。

---

> 以下内容以"我"（Miku_oso）的视角撰写。这是整份协议中为数不多以个人视角书写的部分，我想在这里袒露一些想法。

---

## 1. 起因

大概在 2025 年 10 月，我开始觉得茶馆“等活来找我们”的运营模式不太健康——只要没人找我们，我们就没事做。但当时并没有感到特别紧迫，毕竟还没有出现过断供的问题。

直到断供真的发生了。

这件事让我意识到，茶馆长期依赖少数人来维持运转，一旦这些人不在，整个组织就会停摆。这不是一个可持续的模式。

---

## 2. 为什么用 Git 管理组织

选择 GitHub PR 作为管理工具，核心原因是**权限控制**和**记录可追溯**。

有迹可循，权力行使就不会变成一言堂。每一条规则的修改、每一个决策的诞生，都有 PR 记录作为证据。

至于“管理即代码”这五个字——说实话是临时想出来的。当时只是想借用 Git 的逻辑，而说到 Git 大部分情况下都是代码，所以就随口起了这个名字。现在看来好像跟代码没什么关系，但意思到了就行。

---

## 3. 为什么要求维护者懂点技术

管理人员不具备技术素养，本身就是一种风险来源。

茶馆是一个补档社团——我们处理的是视频、弹幕、脚本、数据。这些东西天然就跟技术绑在一起。如果管理者完全不懂技术，就只能依赖少数懂的人，一旦那些人不在，整个组织就会停摆。

所以我希望维护者们至少能看懂基本的代码逻辑，哪怕只是网页端操作 Git 的程度。不是要求大家都变成程序员，而是希望大家在关键时刻能自己动手，而不是只能等别人来救。

---

## 4. 写作过程中的犹豫

写这份文档最难的地方，是完全不知道它会不会显得太机械、太难理解、太难执行、太麻烦。

直到现在我依然保持怀疑。

但不写，我又觉得这是必须做的事。不是说急需解决，但如果茶馆还想继续往这个方向发展，总得有人把这些东西写下来。不是我，也会是别人，但总得有人先写出来，才能让大家有个方向。

我不认为我提出的方案就是最好的方案。但至少，这是我目前能探索到的最好的方案。

还有一件事一直压在我心里：大家来到茶馆，是为了寻找快乐和意义的。我一直在想，这样激进的改革会不会反而把大家的距离拉得更远？一旦开始"管理"，味道似乎就变了。所以我无数次怀疑过自己——这么做真的合适吗？真的只能这么做吗？

但我还是写了。比起“味道变了”，我更担心的是“茶馆不再是原先那个补档社团了”。

---

## 5. 关于编写者

这份协议的全部内容由 **Miku_oso** 编写，由 **凍り付いた永遠の都** 审查。

冻和我一样，也希望通过彻底改制来改变管理层和整个组织的工作方向。不过他似乎今年三月才渐渐意识到这个问题（我也没仔细问过）。冻对写文档不太擅长，所以 Protocol 的内容由我来执笔，由他来把关。

这个计划一直在“地下”进行，主要是考虑到群里有两位高考生，不想在他们备考期间打扰大家，所以有些先斩后奏了。我知道这样做可能会让人觉得是一言堂——毕竟我平时主要负责写代码，突然跳出来搞治理改革，确实有些奇怪。

---

## 6. 期望

我希望这份文档未来能进化为被所有人认同的共识，而不仅仅是某个人的想法。

未来我也希望它不仅仅是给管理员和维护者看的，也能成为新人了解茶馆运作方式的入口。
