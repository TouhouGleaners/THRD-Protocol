    维护者角色定义：职能分工与架构

    在 THRD | 拾遗梦茶馆，管理不是一种职级，而是一项具体的维护职能。所有的管理者首先是 Maintainer（维护者）。根据个人的精力和专长，维护者主要分为以下职能路径：

    1. 运营维护者 (Operational Maintainer)

    ——“茶馆秩序与内容的守望者”

    运营维护者负责确保茶馆的归档内容准确、社群环境稳定。

    • 核心职责：
      • 内容审核：评估补档视频、弹幕的质量与合规性。
      • 官号维护：负责 B 站等社交平台的发布、宣发与外联。
      • 社群治理：维护日常讨论秩序，处理社群事务。
    • 协作工具：主要通过 GitHub 网页端 进行 Issue 讨论、PR 审核。
    • 定位：确保茶馆的业务运行“有质且有序”。

    2. 技术维护者 (Technical Maintainer)

    ——“茶馆引擎与自动化的捕捉者”

    技术维护者负责茶馆底层设施的稳固，通过自动化工具降低整体维护成本。

    • 核心职责：
      • 逻辑维护：编写与优化补档脚本、TMA 监控逻辑。
      • 工具迭代：维护 danmakusender 等桌面端或网页端工具。
      • 技术保障：解决协作流程中的 Bug 与环境问题。
    • 协作工具：使用 Git 命令行/客户端、Python、Vue 3 等开发环境。
    • 定位：确保茶馆的业务运行“稳健且高效”。

    3. 核心维护者 (Core Maintainer)

    ——“双修”领头人与架构中枢

    核心维护者是具备跨领域理解力的骨干力量，负责在不同职能间建立桥梁。

    • 核心职责：
      • 架构决策：平衡技术实现与运营需求，决定项目优先级。
      • 风险审计：处理重大故障或涉及组织方向的复杂争议。
      • 流程优化：通过对双向逻辑的理解，不断简化协作流程。
    • 要求：同时具备运营视野与基础编程能力。 能够从业务角度审视技术，也能从技术角度支撑业务。
    • 定位：确保茶馆“不偏航”且“可持续”。

    4. 身份转换与动态调整

    鼓励双向演进

    我们认为“复合型”不是一种高低之分，而是一种认知的交汇。

    • 运营向技术靠拢：鼓励运营维护者尝试理解基础代码逻辑。这能让你更直接地参与自动化标准制定，不再受限于“不会改代码”。
    • 技术向运营靠拢：鼓励技术维护者深入理解社群与内容的运作痛点。这能让你写出真正好用、真正解决实际问题的工具，避免“闭门造车”。
    • 动态平衡：你可以根据不同阶段的精力，在技术与运营侧重间灵活切换。

    5. 身份晋升：如何成为核心维护者

    核心维护者（Core Maintainer）不仅是权力的持有者，更是组织架构与技术底线的最后守护人。其产生遵循 “提名-考察-共识” 流程：

    1. 提名：由现任核心维护者发起提名，或由表现卓越的职能维护者发起自荐。
    2. 要求：候选人需具备全栈技术背景（掌握 Python / Vue 3 基础），且在过去 6 个月内持续贡献并严格遵守 Protocol 规范。
    3. RFC 考察：在 THRD-Protocol 开启一个晋升提案 Issue，进入为期 14 天 的公开考察期，听取全体维护者的反馈。
    4. 生效：经现任核心维护者（超过 3 人时需多数通过）最终确认，修改 access-control.md 授权矩阵后正式生效。

    6. 退出与除名机制

    维护者身份的解除分为以下三种情况：

    A. 主动退出 (Voluntary Resignation)

    • 流程：维护者因精力或个人原因拟退出，需发起一个 PR，将其在 access-control.md 中的状态修改为 Emeritus (荣誉成员)。
    • 处理：合并后，相关业务仓库的写权限将予以收回，但其历史贡献将永远保留在 About 页与 Git 记录中。

    B. 自动休眠 (Dormancy)

    • 遵循 权限矩阵 中的休眠保护机制，非活跃 180 天后自动暂时调整权限。

    C. 强制除名 (Removal for Cause)

    • 触发条件：严重违反法律底线、恶意篡改数据、或泄露核心安全密钥。
    • 两步走流程：
      1. 紧急挂起 (Immediate Suspension)：为保护资产安全，任一核心维护者可先行切断该成员的所有写权限，但必须在 24 小时内 在组织内部说明情况。
      2. 最终裁决 (Final Ruling)：挂起后 7 天内，必须发起内部表决。除名需经核心维护者团队多数通过方可转为永久生效。
    • 制衡：若最终裁决未通过，必须恢复其原有权限。所有此类操作必须在 THRD-Protocol 中开 Issue 归档记录。
