    GitHub 标签系统指南 (Labels Reference)

    在 THRD，标签是任务流转的“语言”。请在处理 Issue 和 PR 时，根据以下定义准确标注。

    1. 事项状态 (Status) —— 决定“现在该做什么”

    状态标签用于定义任务当前的生命周期位置。

    标签名称 中文转义与含义 适用场景
    Status: Triage 待分拣 新提交的 Issue，等待核心维护者进行初步评估。
    Status: Needs Investigation 需要调研 方案尚不明确，或 Bug 成因不明，需要进一步测试或查证。
    Status: Needs More Info 需要补充信息 发起人提供的信息不足（如缺少截图、描述模糊），任务暂时挂起。
    Status: Ready for Developer 可开始执行 方案已定，逻辑闭环，维护者可以开始认领并编写代码/文档。
    Status: Mikuoso Ice-Box Mikuoso 的冷藏室 提议暂不执行，但 Mikuoso 认为有参考价值，保留提醒。
    Status: As Designed 符合预期 反馈的问题实际上是系统设计的原本行为，并非 Bug。
    Status: Can't Reproduce 无法复现 维护者无法在相同环境下观察到报告的错误。
    Status: Duplicate 重复项 该 Issue 或 PR 与已有的记录重复。
    Status: Invalid 无效记录 内容不实、恶意灌水或完全偏离组织范畴。
    Status: Won't Fix 不予处理 经讨论决定不采纳该功能或不修复该特定边缘问题。

    2. 事项类型 (Type) —— 决定“这是什么性质”

    类型标签用于区分工作的本质。

    标签名称 中文转义与含义
    Type: Feature 新功能/新规：提议或引入新的核心制度、功能或显著价值。
    Type: Enhancement 优化：对现有功能、性能或用户体验的改进，非新功能。
    Type: Documentation 文档：对 Protocol 或项目说明文字的补充与改进。
    Type: Ethics 伦理与隐私：涉及归档底线、法律合规及个人被遗忘权请求。
    Type: Proposal 独立提案: 临时活动（如周年庆）、新项目立项。仅在 Issue 内流转，不产生物理文件。
    Type: Security 安全：涉及漏洞、凭证泄露或网络风险，需最高优先级关注。
    Type: Breaking change 破坏性变更：会导致不向下兼容的重大调整（需更新配置）。
    Type: Refactor 重构：优化代码/文档结构，不改变外部表现或逻辑功能。
    Type: Performance 性能：侧重于提升运行速度、减少内存占用或响应优化。
    Type: CI/CD 🚧 自动化流：涉及 GitHub Actions、部署脚本等持续集成流程。

    3. 错误分级 (Bug Fix) —— 决定“修Bug的优先级”

    用于标记系统缺陷及其影响程度。

    标签名称 严重程度 定义
    Bug: Critical 紧急/致命 系统崩溃、核心功能失效或数据丢失。必须立即修复。
    Bug: Major 主要/严重 主要功能受损，影响大量用户或核心流程。高优先级修复。
    Bug: Minor 次要/轻微 边缘功能失效或轻微体验问题。不影响核心，有临时绕过方案。
    Bug Fix 常规修复 通用的 Bug 修复标记，请在 PR 中关联对应的 Issue。

    4. 特殊标记与任务 (Special Tags)

    用于快速识别特定属性的任务。

    标签名称 含义 建议
    Quick Fix 快速修正 针对错别字、微小计算错误等。PR 可跳过 Issue 阶段。
    Planning 宏观规划 用于大型变动或提案汇总。核心维护者用其建立“任务索引 Issue”。
    Task: Maintenance 日常维护 依赖库更新、运行环境调整等非功能性维护工作。
    Good First Issue 新手入门 逻辑简单，非常适合新加入或刚转型的维护者作为首个任务。
    Help Wanted 请求协助 负责人遇到瓶颈，需要其他维护者提供额外关注或技术支援。
    Low Priority 低优先级 锦上添花的需求，在精力和资源充沛时再处理。
    Experimentals 实验性 针对不稳定版本或探索性分支的特定议题。
