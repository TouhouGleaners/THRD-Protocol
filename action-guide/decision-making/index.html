    决策流与提案管理 (Decision Making)

    在 THRD | 拾遗梦茶馆，我们通过“异步协作”与“文档对齐”来确保决策的透明与可追溯。我们拒绝碎片化的即时通讯决策，所有的正式决定必须经历 Issue 讨论 到 PR 定稿 的转化。无论你是运营型维护者还是技术型维护者，都通过同一套流程参与组织决策。

    为什么用 PR 管理决策？

    1. 所见即所得：无论 Issue 里聊了多少页，文件里永远只展示最新的、最终的执行方案。
    2. 版本可回溯：通过 Git History，我们可以清晰看到方案在讨论过程中每一个版本的演变。
    3. 强制理性：撰写 Markdown 的过程会迫使发起人理顺逻辑，过滤掉情绪化和不切实际的冲动。

    适用范围说明

    • 本文档适用于：组织级制度变更、重大项目立项、跨职能协作方案、或独立活动提案。
    • 日常任务提醒：对于具体的代码迭代、已知 Bug 修复或纯文字排版优化，请遵循 开发与维护工作流 直接进入任务执行。

    1. 决策工具的分工

    为了兼顾讨论的自由度与结果的严肃性，我们对工具进行明确分工：

    工具 对应阶段 作用
    GitHub Issue 初步想法 / 讨论 用于非结构化的讨论。例如：“我们要不要办周年庆？”“某个脚本是不是该重构？”
    GitHub PR 正式方案 / 评审 用于结构化的定稿。针对具体的文档修改或提案进行逐行评审与修正。
    Git Merge 决策生效 / 归档 核心维护者合并 PR。合并的那一刻，代表决策正式生效，并成为组织历史档案。

    2. 提案分类与路径

    根据事项的性质，维护者应选择不同的管理路径：

    A. 协议修正 (Protocol Modification)

    • 对象：对现有管理制度、技术标准、权限矩阵等长期规范的修改。
    • 路径：Issue 讨论 -> PR 修改文件 -> 核心维护者裁决合并。
    • 意义：这是在更新茶馆的“宪法”，每一处变动都应反映在 docs/ 目录的代码中。

    B. 独立提案 (Proposals)

    • 对象：临时性活动（周年庆）、新项目立项、外部合作等非制度性事务。
    • 路径：直接提交 GitHub Issue，并标注 Type: Proposal。
    • 管理机制：
      1. 讨论：所有脑暴与细节评审均在 Issue 评论区完成。
      2. 规划：由核心维护者通过 Planning 标签下的“任务索引 Issue”进行全盘排期与关联。
      3. 定稿：共识达成后，发起人需更新该 Issue 的首楼描述作为最终执行方案。
      4. 裁决：核心维护者回复“裁决通过”并锁定 Issue 状态。
    • 意义：这是在茶馆的“动态记事本”上规划行动。通过标签检索，Issue 列表本身即成为组织的活动档案库。

    3. 标准提案流程 (RFC 流程)

    RFC（Request for Comments，征求意见稿）是一种通过公开讨论达成共识的流程。

    针对 “路径 A：协议修正” （即需要修改 docs/ 目录下文件的变动），必须遵循以下标准流程：

    第一步：发起 Issue（吹风阶段）

    任何维护者都可以发起 Issue。

    • 目的：试探大家的意向，收集零散的建议。
    • 操作：使用 [讨论] 或 [意向] 标签。

    第二步：提交 PR（起草阶段）

    当想法成熟，发起人需提交一个 Pull Request：

    • 运营维护者：可直接使用 GitHub 网页端的“铅笔”图标进行起草。
    • 技术维护者：建议在本地修改后通过命令行提交。

    第三步：评审与修订（拉锯阶段）

    这是最关键的一步。其他维护者在 PR 的具体行下发表意见。

    • 精准对线：不要说“我觉得方案不行”，而要针对具体条款说“这个方案的时间可能不够”。
    • 持续更新：发起人根据评审意见，直接修改该 PR 中的文件内容，直至达成共识。

    在此期间，发起人根据建议不断完善方案，运营人员可重点关注业务逻辑，技术人员重点关注实现成本。

    第四步：裁决（生效阶段）

    由核心维护者对 PR 进行最终裁决：

    • 批准：任一核心维护者可直接合并。方案正式转化为茶馆的执行指令或历史档案。
    • 拒绝：核心维护者可拒绝合并，PR 保持 open 状态，由拒绝该 PR 的核心维护者打上 Status: Needs Investigation 标签，并在 PR 评论中给出基于本 Protocol 的书面解释。
    • 异议：其他维护者可在拒绝理由发布后 7 天内于 PR 评论中表态反对拒绝。
      • 计票基准：维护者人数以拒绝发生时的名单（参见 权限矩阵）为准，双重身份的维护者只计一票。
      • 通过条件：反对票超过全体维护者的简单多数（即超过半数）。
      • 执行：由任一支持异议的核心维护者执行强制合并。若 7 天内未形成简单多数，拒绝维持。

    关于审计权

    核心维护者的合并权本质上是质量与安全审计。核心维护者应尽量尊重集体共识，仅在提案严重偏离技术标准、法律底线或组织精神时行使拒绝权，并需在 PR 评论中详细说明理由。

    动作 执行者 时限 结果
    批准 任一核心维护者 — 合并生效
    拒绝 任一核心维护者 — PR 保持 open，打标签，书面解释
    异议 其他维护者 7 天 多数反对→强制合并；否则拒绝维持

    4. 紧急例外情况

    紧急避险

    涉及核心服务瘫痪、数据安全等紧急突发状况，负责人可先在群内即时决策并执行。
    记录要求：事故处理完毕后，需在对应仓库开独立 Issue 归档解决过程，并在 Protocol 仓库中登记引用，以便日后检索。
